# 动态网站爬虫工具设计文档

---

## 1. 项目目标

构建一个面向现代动态网站的通用爬虫工具，支持以下能力：

1. 抓取 Vue、React、Next.js 等前端框架渲染后的页面内容。
2. 支持文章页、列表页、商品页等不同页面类型的内容提取。
3. 支持页面渲染抓取、接口监听抓取、登录态复用等多种模式。
4. 支持批量任务、失败重试、断点续跑和结果落盘。
5. 在需要时接入 LLM，对抓取结果做清洗、摘要和结构化整理。

这不是一个“只抓单页并输出 Markdown”的脚本，而是一个可以逐步扩展为生产工具的爬虫框架。

---

## 2. 设计原则

### 2.1 内核优先，Agent 可选

爬虫的核心能力应该先独立成立：

- 能稳定访问页面
- 能判断页面是否加载完成
- 能提取内容
- 能保存结果
- 能处理失败和重试

LLM 或 Agent 只作为后处理能力，用来做摘要、清洗、分类、改写，不应该成为抓取链路的核心依赖。

### 2.2 配置驱动

同一个爬虫框架应尽量通过配置适配不同站点，而不是频繁改代码。

### 2.3 原始数据优先保留

无论最终是否输出 Markdown 或摘要，都建议保留原始 HTML、关键网络响应和基础元数据，便于后续排查、重跑和改进提取规则。

### 2.4 可观测、可恢复

工具必须能回答这几个问题：

- 哪个任务成功了，哪个失败了
- 为什么失败
- 失败后能否重试
- 上次跑到哪了
- 页面结构变动后如何定位问题

## 3. 推荐技术栈

### 3.1 核心抓取

- 动态渲染抓取：`Playwright`
- HTML 解析：`selectolax` 或 `BeautifulSoup`
- 正文抽取：`trafilatura`
- 数据校验：`pydantic`
- HTTP 回退抓取：`httpx`

说明：

- `Playwright` 负责处理动态页面、登录态、滚动和网络监听。
- `trafilatura` 适合文章正文提取，但不适合作为唯一提取方案。
- 对结构化字段提取，仍应保留 CSS Selector、XPath 或 JSON 路径提取方式。

### 3.2 工程能力

- 配置管理：`pydantic-settings` 或 `yaml`
- 日志：`logging` 或 `loguru`
- 本地存储：`SQLite`
- 异步运行：`asyncio`
- CLI：`typer`

### 3.3 可选能力

- 反爬伪装：`playwright-stealth`
- 代理支持：HTTP / HTTPS / SOCKS 代理
- LLM 后处理：OpenAI API 或 LangChain

---

## 4. 系统边界

本工具优先解决以下问题：

1. 抓取动态页面完整内容。
2. 从页面或接口中提取目标数据。
3. 将结果保存为 HTML、Markdown 或 JSON。
4. 支持批量执行、失败重试和调试排查。

本工具暂不默认承担以下职责：

1. 大规模分布式调度
2. 自动绕过高强度验证系统
3. 通用可视化管理后台

这些能力后续可以扩展，但不应影响第一阶段落地。

## 5. 整体架构

```text
用户输入任务 / 配置文件
        |
        v
   [Task Runner]
        |
        |-- 读取任务配置
        |-- 控制并发
        |-- 失败重试
        |-- 断点续跑
        |
        v
   [Fetcher 抓取层]
        |
        |-- Playwright 页面抓取
        |-- 网络请求监听
        |-- 登录态复用
        |-- 自动滚动 / 等待策略
        |
        v
   [Extractor 提取层]
        |
        |-- 原始 HTML 保存
        |-- 正文提取（Trafilatura）
        |-- 结构化字段提取（CSS/XPath/JSON）
        |-- 接口响应提取
        |
        v
   [Pipeline 处理层]
        |
        |-- 清洗
        |-- 去重
        |-- 格式转换
        |-- LLM 后处理（可选）
        |
        v
   [Storage 存储层]
        |
        |-- 本地文件
        |-- SQLite 元数据
        |-- 调试快照
        |
        v
   输出结果 html / markdown / json
```

## 6. 模块设计

### 6.1 Task Runner

负责驱动整套流程。

职责：

- 接收单个或多个 URL 任务
- 管理并发数量
- 执行超时控制
- 记录任务状态
- 执行失败重试
- 支持断点续跑

建议任务状态：

- `pending`
- `running`
- `success`
- `failed`
- `skipped`

建议失败原因分类：

- `timeout`
- `network_error`
- `blocked`
- `login_required`
- `extract_failed`
- `unknown_error`

### 6.2 Fetcher 抓取层

Fetcher 是整个系统的核心模块，负责从目标站点拿到“足够完整”的页面数据。

建议支持三种抓取模式：

1. `rendered_html`
   说明：访问页面，等待渲染完成后获取 HTML。

2. `network_capture`
   说明：监听页面中的 XHR / fetch 请求，直接提取接口返回数据。

3. `hybrid`
   说明：同时保留 HTML 和关键接口响应，兼顾调试和数据完整性。

Fetcher 应支持以下能力：

- 浏览器启动与关闭
- Browser Context 管理
- 自定义 User-Agent
- 自定义 Header / Cookie
- 代理配置
- 登录态 `storage_state` 复用
- 超时设置
- 多种等待策略
- 自动滚动
- 页面截图
- 页面 HTML 快照保存

### 6.3 Extractor 提取层

Extractor 不应只依赖 `trafilatura`，而应按页面类型选择提取策略。

建议支持以下提取方式：

1. 正文提取
   场景：新闻、博客、帮助文档、长文章页面
   方案：`trafilatura`

2. 结构化字段提取
   场景：商品页、详情页、个人主页
   方案：CSS Selector / XPath / 正则 / JSON 路径

3. 列表提取
   场景：搜索结果页、文章列表页、商品列表页
   方案：针对重复节点批量提取字段

4. 接口响应提取
   场景：前端页面只做展示，真实数据都来自 API
   方案：监听网络请求并保存 JSON 响应

### 6.4 Pipeline 处理层

负责把原始结果变成最终可使用的数据。

能力建议：

- 文本清洗
- 字段标准化
- 内容去重
- 结果格式转换
- 长文本切片
- LLM 摘要与分类

建议注意：

- LLM 输出不应覆盖原始数据
- 原始 HTML 和原始 JSON 必须保留
- 所有清洗结果都应可追溯到原始输入

### 6.5 Storage 存储层

建议将存储拆成两层：

1. 文件存储
   保存原始 HTML、截图、Markdown、接口 JSON

2. 元数据存储
   使用 SQLite 记录任务执行信息和文件路径

建议保存字段：

- `task_id`
- `url`
- `final_url`
- `status`
- `mode`
- `http_status`
- `title`
- `content_hash`
- `started_at`
- `finished_at`
- `error_type`
- `error_message`
- `html_path`
- `markdown_path`
- `json_path`
- `screenshot_path`

## 7. 页面加载完成策略

这是动态爬虫中最重要的设计点之一，不能只依赖 `networkidle`。

建议支持以下等待方式：

1. `domcontentloaded`
   适合首屏结构很快出现的页面。

2. `load`
   适合资源不多、页面比较传统的站点。

3. `networkidle`
   适合无持续轮询的页面，但不应作为唯一默认策略。

4. `selector`
   等待特定元素出现，例如文章正文容器、商品标题、评论列表。

5. `response`
   等待某个关键接口返回成功。

6. `scroll_until_stable`
   用于懒加载、无限滚动场景。

建议默认逻辑：

1. 先 `domcontentloaded`
2. 再等待关键 selector 或短时间静默
3. 如配置要求则执行滚动
4. 最后再抓取 HTML 和网络响应

## 8. 配置设计

建议使用 YAML 或 JSON 管理站点规则。

示例：

```yaml
site_name: example_news
allowed_domains:
  - example.com

fetch:
  mode: rendered_html
  headless: true
  timeout_ms: 30000
  wait_until: domcontentloaded
  wait_selector: article
  auto_scroll: false
  proxy: null
  storage_state: null

extract:
  type: article
  title_selector: h1
  content_selector: article
  use_trafilatura: true

output:
  save_html: true
  save_markdown: true
  save_json: true
  save_screenshot: true

retry:
  max_retries: 3
  retry_on:
    - timeout
    - network_error

limits:
  concurrency: 3
  delay_seconds: 2
```

这样做的好处是：

- 新增站点时优先改配置，不急着改代码
- 便于多站点复用
- 便于后续做批量任务模板

## 9. 任务模型设计

建议一个任务至少包含：

```json
{
  "task_id": "uuid",
  "url": "https://example.com/article/123",
  "site_name": "example_news",
  "fetch_mode": "rendered_html",
  "output_format": ["html", "markdown", "json"],
  "priority": 5,
  "retry_count": 0
}
```

如果后续要支持批量采集，应再加上：

- 来源渠道
- 创建时间
- 计划执行时间
- 上次执行时间
- 去重键

## 10. 去重与增量抓取

这是很多爬虫工具后面补起来很痛苦的一部分，建议一开始就设计。

### 10.1 去重策略

至少支持以下两层：

1. URL 去重
   防止同一链接重复抓取。

2. 内容去重
   对正文或结构化结果计算 `content_hash`，防止页面 URL 变了但内容重复。

### 10.2 增量抓取

适合列表页或时间线抓取。

可选策略：

- 按发布时间截止
- 按列表中首次遇到已抓取链接停止
- 按内容哈希去重

---

## 11. 错误处理与重试策略

建议区分“可以重试”和“不建议重试”的错误。

可重试：

- 网络超时
- 临时 DNS 错误
- 浏览器偶发崩溃
- 页面短时加载失败

不建议直接重试：

- 明确需要登录
- 明确被封禁
- 选择器失效
- 页面已删除

建议重试策略：

- 最大重试次数：`3`
- 重试间隔：指数退避
- 每次重试前重新创建 page，必要时重新创建 context

---

## 12. 调试与可观测性

生产中最常见的问题不是“代码报错”，而是“页面结构变化后抓不到东西”。

建议默认保留以下调试信息：

- 抓取前后的关键日志
- 最终页面 URL
- 页面标题
- 页面截图
- 最终 HTML 快照
- 命中的关键网络请求
- 提取结果摘要

建议日志中包含：

- `task_id`
- `url`
- `site_name`
- `mode`
- `status`
- `duration_ms`
- `error_type`

---

## 13. 登录态与反爬处理

### 13.1 登录态支持

对于需要登录的网站，优先采用以下方式：

1. 手动登录一次
2. 保存 `storage_state.json`
3. 运行任务时复用登录态

注意事项：

- 登录态文件应视为敏感信息
- 应支持按站点区分多个状态文件
- 状态失效后要能快速重新录制

### 13.2 反爬处理

建议作为可选能力接入，而不是默认强绑定。

可选项：

- `playwright-stealth`
- 随机化 User-Agent
- 控制访问频率
- 代理池
- 降低并发

需要强调：

- 不建议把“绕过一切防护”当作系统目标
- 合法合规和目标站点规则需要单独评估

---

## 14. 输出格式设计

建议至少支持三类输出：

### 14.1 HTML

用途：

- 原始归档
- 调试排查
- 二次解析

### 14.2 Markdown

用途：

- 文章内容沉淀
- 知识库入库
- 便于人工阅读

### 14.3 JSON

用途：

- 程序消费
- API 输出
- 后续数据分析

推荐做法：

- 同一任务可同时输出多种格式
- Markdown 和 JSON 都应关联原始 HTML

---

## 15. LLM / Agent 的正确位置

LLM 不建议直接承担“抓网页”职责，而应承担以下工作：

1. 清洗脏乱 Markdown
2. 长文摘要
3. 结构化信息抽取
4. 标签分类
5. 输出用户需要的最终文档

推荐流程：

```text
URL
  -> 爬虫抓取
  -> 原始 HTML / JSON 保存
  -> 提取正文或结构化字段
  -> 可选 LLM 后处理
  -> 输出最终 Markdown / JSON
```

是否使用 LangChain 取决于你的最终目标：

- 如果只是做爬虫工具：不必强依赖 LangChain
- 如果要做“抓取 + 总结 + 问答”助手：再接 Agent 更合适

## 17. MVP 实现顺序

为了避免一开始做得太大，建议按下面顺序推进：

### 第一步：先做最小可运行内核

目标：

- 输入一个 URL
- 用 Playwright 抓取渲染后 HTML
- 保存 HTML、截图、基础日志

### 第二步：补提取能力

目标：

- 文章页输出 Markdown
- 结构化页面输出 JSON
- 支持 selector 配置

### 第三步：补任务系统

目标：

- 批量任务执行
- 状态记录
- 失败重试
- SQLite 落盘

### 第四步：补高级抓取能力

目标：

- 自动滚动
- 网络接口捕获
- 登录态复用
- 代理支持

### 第五步：最后再接 LLM

目标：

- Markdown 清洗
- 摘要生成
- 按模板输出最终文档

## 18. 当前方案相比初稿的主要升级点

这一版相对于“Playwright + Trafilatura + LangChain Tool”的初稿，主要提升在于：

1. 从单页 demo 升级为工程化框架设计。
2. 明确把抓取内核和 LLM 后处理解耦。
3. 增加任务系统、状态管理、失败重试和断点续跑。
4. 不再只依赖正文提取，补上结构化提取和接口抓取。
5. 增加配置体系、存储设计和调试能力。
6. 为后续批量任务和多站点扩展留出清晰边界。

---

## 19. 最终建议

如果你的目标是“做一个真正能长期用的动态网站爬虫工具”，推荐你采用下面的核心路线：

1. 以 `Playwright Fetcher` 为内核。
2. 以“配置驱动的提取器”适配不同站点。
3. 以 `SQLite + 文件存储` 管理结果和任务状态。
4. 以 `HTML / Markdown / JSON` 作为标准输出。
5. 将 LLM 放在后处理阶段，而不是放在抓取链路正中央。

这样做的好处是：

- 先解决“能稳定抓到”
- 再解决“抓得准”
- 最后解决“如何更智能地整理和使用”

这条路线更稳，也更容易扩展成你后面真正想要的工具。
